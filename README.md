# Rya: Doubly Oblivious MapReduce

## 🗂️ Project Structure & Execution Pipeline

The execution must follow a strict cryptographic boundary (Trusted Client vs. Untrusted Server). Based on the project structure, the modules are organized as follows:

* **`Step0_Client_Data_Process` (Trusted Client)**: Prepares and maps data before sending it to the server, and demaps the results.
* **`Rya_Example_NO_visualization` (Untrusted Server - High Performance)**: The streamlined, performance-oriented implementation of the Rya protocol. Stripped of all logging to strictly test execution speed and security.
* **`Rya_Example_with_Visualization` (Untrusted Server - Debug/Educational)**: Identical protocol logic but includes verbose console logging, step-by-step Butterfly Network routing prints, and OBIN state visualizations.
* **`Python_Generate_Data` (Testing Tool)**: Generates tightly controlled mock datasets to test the system's obliviousness against distribution-based leakages.
* **`Rya`**: Core library or shared dependencies for the protocol.

### `Step0_Client_Data_Process` (Trusted Client)
**⚠️ MANDATORY FOR TEXT (STRING) DATA**
The Rya Server modules strictly compute on integers (`Int`) **(from 0 to numUniqueKeys)** to prevent side-channel leaks caused by string comparisons, and help OBIN align perfectly after OBINSUM (refer to our paper). 
* **PreProcess**: Converts your raw datasets or integers datasets that don't fulfill the requirement of **(from 0 to numUniqueKeys)** (e.g., `data.txt`) into dense integer arrays (`_Rya_Input.txt`) and generates a *secret local dictionary mapping* (`_Rya_Mapping.csv`).
* **AfterProcess**: Once the Server finishes computation, this mode reads the Server's output *numerical histogram* (`Int`) **(from 0 to numUniqueKeys)** and translates it back into the *histogram* of original data using the secret dictionary (`_Rya_Mapping.csv`) generated by PreProcess.

## 📊 Obliviousness Testing (Data Generation)
If you want to verify that the Rya protocol successfully hides access patterns and is impervious to distribution-based leakages, navigate to the `Python_Generate_Data` directory. 

By running `bash generate_data.sh`, you can generate identically sized test files (e.g., 50MB) with wildly different statistical distributions (100% skewed identical items vs. 100% uniform random items). A truly oblivious system will process these distinct files with identical execution times and memory traces.

## 🚀 Recommended Workflow
To test the entire lifecycle on a dataset like `Example.txt`:

1. **Test Data Prep (Optional)**: Go to `Python_Generate_Data` and generate identically sized test files with different distributions to benchmark obliviousness.
2. **Client Prep**: Go to `Step0_Client_Data_Process`. Run `PreProcess` on your target dataset (e.g., `Example.txt` or `random_string.txt`). Take the generated `_Rya_Input.txt`.
3. **Server Computation**: Move the `_Rya_Input.txt` into `Rya_Example_NO_visualization` (for speed) or `Rya_Example_with_Visualization` (for debugging). Run the module to generate `_Rya_Output.txt`.
4. **Client Decode**: Bring the `_Rya_Output.txt` back to `Step0_Client_Data_Process`. Run `AfterProcess` to recover the output of original data `_Rya_Output_Final.txt` utilizing your secret dictionary `_Rya_Mapping.csv` generated by `PreProcess`.
