# Rya: Doubly Oblivious MapReduce (Visualization Prototypes)

This repository contains the interactive **visualization prototypes** for the **Rya** protocol. To facilitate understanding, debugging, and academic review, the core components of the Doubly Oblivious MapReduce system have been decoupled into distinct, sequential steps.

These modules are heavily instrumented with *console printouts*, allowing you to trace the exact memory flows, sorting networks, and bin-packing states step-by-step.

## 🗂️ Project Structure & Execution Pipeline

The execution must follow a strict cryptographic boundary (Trusted Client vs. Untrusted Server).

### `Step0_Client_Data_Process` (Trusted Client)
**⚠️ MANDATORY FOR TEXT (STRING) DATA**
The Rya Server modules strictly compute on numerical identifiers (`Int`) to prevent side-channel leaks caused by string comparisons, and help OBIN align perfectly after OBINSUM (refer to our paper). 
* **PreProcess**: Converts your raw string datasets (e.g., `data.txt`) into dense integer arrays (`_input_int.txt`) and generates a *secret local dictionary mapping* (`_mapping.csv`).
* **AfterProcess**: Once the Server finishes computation, this mode reads the Server's output *numerical histogram* (`Int`) and translates it back into a human-readable *text histogram* (`String`) using the secret dictionary (`_mapping.csv`) generated by PreProcess.

### `Step1_Record_Distribution` (Untrusted Server)
The first phase of the Rya protocol. This module takes the mapped integers and obliviously distributes the records, preparing the workloads for parallel processing without revealing data distribution or partition affinities.

### `Step2_Histogram_Creation` (Untrusted Server)
The core aggregation phase. It utilizes the **Butterfly Network (Bin-and-Sum)** and **Oblivious Bin-Packing (OBIN)** to securely count and route identical keys into a unified local histogram without relying on data-dependent memory accesses. 
* *Input: Integer array from Step 0 or Step 1.*
* *Output: An aggregated Integer Histogram.*

### `Step3_Balanced_Allocation` (Untrusted Server)
The final MapReduce shuffle and allocation phase. It takes the histograms and obliviously balances the workload across reducers, ensuring that the volume of data assigned to any specific node remains completely hidden from adversaries.

### `Oblivious_Sort` (Standalone Utility)
A dedicated module containing pure implementations of our oblivious sorting primitives (e.g., **Bitonic Sort** and **Oblivious Bucket Sort**) used inside the RYA protocol.

## 🚀 Recommended Workflow

To test the entire lifecycle on a dataset like `Example.txt`:

1. **Client Prep**: Go to `Step0_Client_Data_Process`. Run `PreProcess` on `Example.txt`. Take the generated `Example_input_int.txt`.
2. **Server Computation**: Move `Example_input_int.txt` into `Step2_Histogram_Creation` (or Step 1). Run the module to generate `Example_output_int.txt`.
3. **Client Decode**: Bring `Example_output_int.txt` back to `Step0_Client_Data_Process`. Run `AfterProcess` to recover the final `Example_output_string.txt`.
